# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

**Root (runs both packages):**
```bash
bun run dev          # Start backend + frontend with hot reload
bun run build        # Build frontend (Vite) then backend (Bun)
bun run lint         # Biome lint on both packages
bun run typecheck    # TypeScript strict check on all packages
bun run test         # Run tests across all packages
```

**Backend:**
```bash
cd backend
bun run dev                    # Hot reload server on :4200
bun run migrate:generate       # Generate Drizzle migrations from schema changes
bun run migrate:run            # Run pending migrations
bun test src/path/to/file.test.ts   # Run a single test file
```

**Frontend:**
```bash
cd frontend
bun run dev          # Vite dev server on :5173 (proxies /api → :4200)
bun test src/path/to/file.test.tsx   # Run a single test file
```

## Architecture

Hexagonal architecture with three layers:

- **Domain** (`backend/src/domain/`): Pure entities + repository/gateway port interfaces. No framework deps.
- **Application** (`backend/src/application/`): Use cases that accept port interfaces as constructor args. Business logic lives here.
- **Infrastructure** (`backend/src/infrastructure/`): Drizzle repository implementations, HTTP gateways (Loki, git), crypto utilities, schedulers.
- **HTTP** (`backend/src/http/`): Hono route handlers — thin layer that calls application use cases and serializes responses.

Dependency injection is assembled in `backend/src/bootstrap/compose.ts`. All repositories are injected at startup; use cases are instantiated per-request in route handlers with their dependencies.

**Backend startup sequence** (`backend/src/index.ts`):
1. Validate env (Zod schema in `config/env.ts`)
2. Auto-run Drizzle migrations
3. Seed admin user if `users` table is empty
4. Build DI container → start Hono server
5. Start Loki integration sync scheduler + marketplace-source sync scheduler

**Frontend** is a React 18 SPA with React Router v7. Auth state lives in `AuthContext` which calls `/api/auth/me` on mount. The `apiFetch` client in `lib/api.ts` handles credentials and 401 → logout redirect. All API calls go through the namespaced `api.*` object exported from `lib/api.ts`.

Sync-status fan-out uses dedicated context providers (`IntegrationsHealthContext`, `MarketplaceSourcesHealthContext`) that wrap `AppShell` in `ProtectedRoute`. They poll their respective endpoints and feed status badges in the sidebar. When you add a new background-syncing entity, follow this pattern instead of fetching status inside each component.

## Key Technical Choices

- **Runtime & package manager**: Bun (not Node/npm). Use `bun` for all commands.
- **Backend framework**: Hono (lightweight, edge-compatible)
- **ORM**: Drizzle — schema is the source of truth in `backend/src/db/schema.ts`. Run `migrate:generate` after any schema change.
- **Linter/formatter**: Biome (not ESLint/Prettier). Config in `biome.json` per package.
- **CSS**: Tailwind CSS v4 via Vite plugin
- **Tests**: Bun test runner. Frontend uses happy-dom (configured in `test-setup.ts`).
- **Auth**: JWT sessions (cookie `session`), separate Bearer tokens for OTLP ingestion

## Environment

Copy `.env.example` to `.env` and fill in:
- `DATABASE_URL` — PostgreSQL connection string
- `JWT_SECRET` — min 32 chars
- `ADMIN_EMAIL` + `ADMIN_PASSWORD_INITIAL` — bootstrapped on first empty DB run
- `PUBLIC_BASE_URL` — used in onboarding instructions

For local dev with Docker: `docker compose -f docker-compose.dev.yml up -d` starts PostgreSQL on `0.0.0.0:5432`.

## Database

Schema: `backend/src/db/schema.ts` (single file, all tables).  
Migrations: `backend/src/db/migrations/` (auto-generated by Drizzle, committed to repo).  
After changing the schema, always run `bun run migrate:generate` from `backend/` — do **not** hand-author migration files or edit `meta/_journal.json` manually.

**Why this matters:** Drizzle's runtime migrator (`drizzle-orm/postgres-js/migrator`) decides whether to apply a migration by comparing its journal `when` value to the **max** `created_at` in `__drizzle_migrations`. If you add a new entry with a `when` value `<=` any already-applied migration's `when`, it will be **silently skipped forever** (no error, no warning). This bit us once — three migrations had May-2025 `when` values smuggled into a journal whose head was already in March 2026, so they never ran in production. If you ever must add a migration without `migrate:generate`, the `when` value MUST be strictly greater than every existing entry's `when`. Prefer `Date.now()` at file-creation time. Also keep migration SQL idempotent (`CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`) so re-running on a partially-applied DB is safe.

**Array binding in `sql\`...\`` templates:** never write `ANY(${jsArray}::text[])` or `unnest(${jsArray}::text[], ...)` — postgres-js flattens the JS array into scalar params, so the cast runs against a single text value and the query throws (`malformed array literal`). Use Drizzle's query builder with `inArray(col, jsArray)` from `drizzle-orm` instead (see `DrizzlePluginSkillRepository.deleteByPlugins`). This has bitten us twice in skill-repository methods — both times the failure was silently swallowed by an outer try/catch and left rows in an inconsistent state, not a visible crash.

## Features

- **Skills** (`/skills`, `SkillsTablePage`): The core domain — every `claude_code.skill_activated` event ingested via OTLP is bucketed by skill name and plugin. Skills are derived from events (no manual create); the `skills` table is upserted on ingest. Status (`to_review` / `approved` / `removed`) is inherited from the owning plugin's marketplace status. New skills land in `to_review` by default; bundled built-ins (`skillSource === "bundled"`) display as `approved` via the `defaultBundledStatus` helper.
- **Integrations** (`settings/integrations`, `IntegrationsPage`): Loki-backed pull ingestion. Each integration has a `url`, `lokiQuery`, and `syncIntervalMs`. The scheduler in `infrastructure/scheduler/sync-scheduler.ts` polls Loki and feeds events into the same pipeline as OTLP push. Encrypted basic-auth credentials via `auth_password_encrypted`.
- **Tokens** (`/tokens`, `TokensPage`): Bearer tokens minted for OTLP ingestion. Stored hashed; revocation via `revoked_tokens` JTI list. Separate from session JWTs.
- **Marketplace Sources** (`marketplace-sources`): Git-backed plugin registries. Each source has a `gitUrl`, optional encrypted `accessToken`, `branch`, and configurable `syncIntervalMs`. On sync, the gateway fetches `marketplace.json` from the git host and upserts marketplace + plugin records. Schedulers run per-source at their configured interval.
- **Plugins**: Synced from marketplace sources. Read-only from the UI (`PluginsPage`).
- **Audit Log**: All write operations emit audit events (`AuditLogPage`).

## OTLP Ingestion

The telemetry endpoint (`POST /api/v0/telemetry/v1/logs`) accepts OTLP HTTP/JSON format. Parsing logic is in `backend/src/parsers/`. Authenticated via Bearer token (ingestion tokens, not session JWTs). The endpoint responds in OpenTelemetry `PartialSuccess` format.

## Adding a New Feature

Typical flow for a new backend feature:
1. Add domain entity/port to `domain/`
2. Add use case(s) to `application/`
3. Add Drizzle repository to `infrastructure/repositories/`
4. Wire it in `bootstrap/compose.ts`
5. Add Hono route handler to `http/`
6. Register the route in `app.ts`

Frontend: add page to `pages/`, add API method to `lib/api.ts`, add route to `App.tsx`.

For features that call external services: also add a gateway port interface to `domain/ports/` and an HTTP gateway implementation to `infrastructure/gateways/`. Inject the gateway into use cases alongside repositories.

## Frontend Design System

Minimal in-house primitives in `frontend/src/components/ui/`. Compose with `cn()` (clsx + tailwind-merge) — no Radix, shadcn, or cva.

**Folder layout under `frontend/src/components/`:** `ui/` holds generic primitives (Button, Table, …); the sibling folders `audit/`, `charts/`, `integrations/`, `layout/`, `marketplaces/`, `plugins/`, `skills/` hold domain-specific composites. Put new domain UI in the matching folder; reach for `ui/` only when the component is genuinely generic.

- **Tokens**: CSS custom properties + `@utility` classes (`btn-primary`, `badge-*`, `nav-active`, `dot-grid`) live in `src/index.css`. Add new tokens there, not inline.
- **Primitives**: `Button`, `Input`, `Select`, `FormField`, `Card`, `PageHeader`, `Table` (+ `THead`/`TBody`/`TR`/`TH`/`TD`/`EmptyRow`). Import from the barrel `@/components/ui`.
- **Domain primitives**: `StatusBadge`, `SearchBar`, `MultiSelect`, `StatusFilter`, `SegmentedControl`, `Drawer`, `Menu`/`MenuItem`/`ConfirmMenuItem`, `Sparkline` — kept as-is.
- **Shared classes**: `_styles.ts` exports `INPUT_BASE`, `DROPDOWN_PANEL`, `MENU_ITEM_BASE` for primitives that own their own positioning logic (SearchBar, MultiSelect, Menu).
- **Rule of thumb**: prefer the primitive over inline Tailwind for buttons/inputs/cards/tables/page headers. Reach for `cn()` when composing variants. Tests use `bun:test` + `renderToString` from `react-dom/server` (see `Button.test.tsx` as a template).
